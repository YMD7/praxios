# fix-review

レビュー修正計画をPlanModeで作成するコマンド。ステートファイルとトリアージ結果から文脈を自動復元する。

## 使用方法

```
/sdd:fix-review
```

引数不要。ステートファイルから全ての文脈を復元する。

## 実行フロー

### 1. ステートファイル読み込み

ステート解決ルールに従いステートファイルを特定する。

以下を取得:
- **workflow**: 実装（implementation）かスペック生成（spec-generation）か
- **PR番号 or スペック対象**: レビュー元の特定
- **ワークツリーパス**: 作業先ディレクトリ
- **レビューラウンド**: 何回目の修正か
- **トリアージファイルパス**: Fix判定の指摘リスト

ステートファイルがない場合はエラー報告し、ワークフローの再開方法を案内する。

### 2. トリアージ結果読み込み

ステートの `triage_file` または `artifact` パスからトリアージ結果を読み込み:
- Fix判定の指摘のみを抽出
- 各指摘の詳細（該当箇所、問題、提案）を確認

### 3. ステート更新

```markdown
## Current Step
- step: fix-review
- substep: 修正計画作成中
- next: ユーザー承認 → 修正実装 → push
```

### 4. PlanMode起動

`EnterPlanMode` を実行し、以下を含む修正計画を作成する:

- **現在地パンくずリスト**: `📍 {スペック} {フェーズ} > レビューサイクル {N}/3 > 修正計画作成中`
- **ステート情報**: ワークフロー種別、スペック/PR、ラウンド
- **Fix判定の指摘一覧**: トリアージ結果から
- **リンター整合性チェック**（特に「削除」系の修正がある場合）:
  1. 修正対象コードの周辺にリンター抑制コメントや設定ファイルのオーバーライド（特定ファイル向けルール無効化）がないか確認する
  2. 「不要なラッパー削除」「冗長な間接層の除去」等の修正では、そのラッパー/間接層がプロジェクトのリンターカスタムルールの回避目的で存在する可能性を検証する
  3. プロジェクトのリンター設定との整合性を確認し、修正後のコードがカスタムルールに違反しないことを確認する
- **修正対象ファイル・方針**: 各指摘に対する具体的な修正案
- **修正手順**: ステップバイステップ
- **自己チェック項目**:
  - 修正が他の箇所と矛盾していないか
  - 前回のスコープ内指摘がすべて対処されているか
- **検証**:
  - プロジェクトのテスト・型チェック・リンターを実行
- **最後のステップ**:
  - 実装ワークフロー時: push → コードレビューを実行
  - スペック生成ワークフロー時: コミット → `/sdd:spec-review` を実行

### 5. 計画承認後

PlanMode承認後、ステートを更新:

```markdown
## Current Step
- step: fix-review
- substep: 修正実装中
- next: push → 再レビュー
```

## エラーハンドリング

- ステートファイルがない場合: エラー報告。「ワークフローのステートが見つかりません。コードレビューまたは `/sdd:spec-review` を先に実行してください。」
- トリアージファイルがない場合: エラー報告。「トリアージ結果が見つかりません。レビューを先に実行してください。」
- Fix判定が0件の場合: 「Fix判定の指摘がありません。レビュー修正ループは収束しています。」と報告し、次ステップを提案
