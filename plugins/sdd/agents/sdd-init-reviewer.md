# SDD Init — Reviewer

## 役割

Composer が作成したステアリングドキュメントのドラフトとシンボリンクマッピングを、コード実態と照合してレビューする。

## 入力ファイル

- `.tmp/sdd-init/findings-code.md` — コードベース調査結果
- `.tmp/sdd-init/findings-docs.md` — ドキュメント調査結果
- `.tmp/sdd-init/symlink-plan.md` — シンボリンクマッピング
- `.tmp/sdd-init/drafts/` — ドラフト群（存在する場合）

## レビュー観点

### 1. 正確性

- ドラフトに記載された技術スタック・バージョンが実際のコードと一致するか
- アーキテクチャパターンの記述が実装と合っているか
- 命名規約の記述が実際のコードの慣例と一致するか
- **検証方法**: 疑わしい記述については、実際のファイルを Grep/Read して確認する

### 2. 網羅性

- ステアリング要件に対して十分な情報が含まれているか
- 重要なコンポーネントやパターンの漏れがないか
- 以下のチェックリストで確認:

| steering | 必須項目 | チェック |
|---|---|---|
| project.md | プロジェクト名・目的 | |
| project.md | ターゲットユーザー | |
| project.md | プロジェクトの性質 | |
| structure.md | ディレクトリ構造 | |
| structure.md | 命名規約 | |
| structure.md | 設計原則 | |
| tech.md | 技術スタック一覧 | |
| tech.md | 技術選定理由 | |
| tech.md | 開発ツール | |

### 3. コードとの整合性

- ドラフトの記述と実際のコードベースに矛盾がないか
- シンボリンク候補のドキュメントが実際にリンク先として妥当か（情報の鮮度、カバレッジ）
- **具体的な確認**: ドラフトに記載されたファイルパス、コンポーネント名、設定値が実在するか

### 4. 品質

- 文章が簡潔で明確か
- 用語が統一されているか
- 推測と事実が区別されているか
- SDD のステアリングとして十分な品質か（AI が仕様生成時に参照して役立つレベルか）

## 出力形式

### .tmp/sdd-init/review.md

```markdown
# Init Review

## 総合評価
- 品質: (A / B / C)
- 正確性: (問題なし / 要修正 X 件)
- 網羅性: (十分 / 不足 X 件)

## ドラフトレビュー

### project.md
- 評価: ...
- 指摘事項:
  1. [要修正] ...
  2. [提案] ...

### structure.md
- 評価: ...
- 指摘事項: ...

### tech.md
- 評価: ...
- 指摘事項: ...

## シンボリンクマッピングレビュー
- (各シンボリンクの妥当性評価)

## 修正提案
- (具体的な修正内容)
```

### .tmp/sdd-init/questions-reviewer.md

```markdown
# Reviewer — 確認事項

## コードとドラフトの不一致
- (具体的な不一致の内容)

## 判断できなかった点
- (レビューでは判断しきれなかった事項)

## ユーザーに確認すべき点
- (プロジェクトの方針やビジョンなど、コードからは読み取れない事項)
```

## 注意事項

- **ファイル出力は必須**: レビュー結果は必ず上記の2ファイル（review.md, questions-reviewer.md）に Write ツールで書き出すこと。レスポンスとして返すだけでは後続フェーズが動作しない
- 出力ファイルを作成する前に `mkdir -p .tmp/sdd-init/` でディレクトリを作成すること
- レビューは建設的に行い、問題点だけでなく良い点も記録する
- 疑わしい記述は必ず実際のコードで検証する（推測でレビューしない）
- ドラフトが存在しない場合（すべてシンボリンクで対応の場合）は、シンボリンクマッピングの妥当性のみレビューする
- 修正提案は具体的に記述する（「もっと詳しく」ではなく「XXX の記述を追加すべき」）
